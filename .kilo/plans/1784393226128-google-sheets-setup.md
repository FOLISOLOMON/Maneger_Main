# Veloura Manager — Google Sheets Backend Setup

This document explains how the **Google Apps Script (`Code.gs`)** backend maps to the
previous Supabase schema, whether the two match, and gives you the **exact tables
(sheets) and every column** you must create in your Google Sheet so the app works.

---

## 1. Does `Code.gs` match the Supabase database?

**Short answer: Not yet.** `Code.gs` is a *generic CRUD scaffold* whose `SHEET_DEFINITIONS`
already use the **same column names as the Supabase tables** (good), but it is missing
the **business logic** that currently lives in Supabase RPCs, and it has no table for
`id_sequences` (Apps Script replaces that with its own counter).

### What already matches (field names)
The Apps Script sheet definitions mirror the Supabase column names for all 10 entity
sheets: `Settings`, `Suppliers`, `InventoryBatches`, `Products`, `Customers`, `Sales`,
`Expenses`, `WalletTransactions`, `Notifications`, `ActivityLogs`.

### What does NOT match / is missing
| Area | Supabase | `Code.gs` today | Fix needed |
|------|----------|-----------------|-----------|
| ID generation | `generate_business_id()` backed by `id_sequences` table | `PropertiesService` script counter (`generateBusinessId`) | OK — Apps Script uses counters instead of a table |
| `record_sale` RPC | validates stock, decrements product, recomputes batch aggregates (revenue/profit/roi/completion/status) | only decrements `current_stock` | Must re-implement recompute in `Code.gs` |
| `close_batch` RPC | finalizes batch + allocates net profit into Needs/Savings/Growth wallets | not implemented | Must implement in `Code.gs` |
| `void_sale` RPC | restores stock + recompute | not implemented | Must implement in `Code.gs` |
| `recompute_batch` RPC | recomputes stored batch aggregates | not implemented | Must implement in `Code.gs` |
| Joins (`supplier`, `product`, `customer`, `batch`) | Supabase returns nested related rows | Apps Script returns flat rows | Add a small join layer so API returns related objects the UI expects |
| RLS / single-row settings | enforced in DB | not needed (single-tenant app) | Enforce "one Settings row" in script if desired |

**Bottom line:** Create the 10 sheets below with the exact columns. Then the connection
layer (`src/lib/sheets.ts`) + the extended `Code.gs` API will let the app read/write the
same shapes the React UI already expects.

## 2. Google Sheet structure

Create **one Google Sheet** (one spreadsheet file). Inside it, create **10 tabs (sheets)**.
The tab name **must exactly match** the sheet name in the table below (these are the names
`Code.gs` looks up). Put the **column headers in row 1** exactly as listed, in the given
order. The Apps Script `initializeVelouraSheets()` function can also generate these for you
via the "Veloura Manager → Initialize sheets" menu.

> Tip: Every sheet needs an `id` column as the first column (the business ID, e.g.
> `BAT-000001`). These IDs are generated by Apps Script — do **not** type them manually.

## 3. Quick setup steps in Google Sheets
1. Create a new Google Sheet.
2. Create the 10 tabs named exactly as above.
3. In row 1 of each tab, type the column headers in the exact order listed.
4. Paste `Code.gs` into **Extensions → Apps Script**, then run
   `initializeVelouraSheets()` (or use the **Veloura Manager** menu) to auto-create them.
5. Deploy `Code.gs` as a **Web App** (Execute as: Me, Access: Anyone) and copy the URL.
6. Put that URL in the app's `VITE_SHEETS_API_URL` (see connection section).

---

## 4. Connecting the app to the Google Sheet (what gets changed)
The React app currently talks to Supabase through `src/lib/supabase.ts` and
`src/services/api.ts`. To switch to the Sheet:

- **`src/lib/supabase.ts`** → replaced by a Sheets client (`src/lib/sheets.ts`) that
  `fetch()`-es the deployed Apps Script web-app URL (configured via `VITE_SHEETS_API_URL`).
- **`src/services/api.ts`** → every function keeps its **current name and return shape**
  (the UI depends on it) but now calls the Sheets JSON API instead of `supabase.from(...)`
  / `supabase.rpc(...)`.
- **`src/services/id.ts`** → removed (`generateBusinessIdRpc`); IDs are now produced by
  `Code.gs` on create.
- **`src/hooks/queries.ts`** → no change (it only calls `api.*`).
- **Business logic** (`recordSale`, `voidSale`, `closeBatch`, `recomputeBatch`,
  `createExpense`) is re-implemented in `Code.gs` so the stored batch aggregates stay
  correct. Derived UI values (wallet balances, KPIs, batch health) remain **computed in the
  frontend** (`src/services/calculations.ts`) — only source rows are stored in the sheet.
- **Relations/joins**: the `Code.gs` API returns related objects inline (e.g.
  `batch.supplier.supplier_name`, `sale.product.product_name`) to match what the pages
  currently read from Supabase nested results.

### Environment variables
Remove: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`.
Add: `VITE_SHEETS_API_URL` (the deployed web-app URL). Optionally
`VITE_SHEETS_API_KEY` (a shared secret the Apps Script verifies).
